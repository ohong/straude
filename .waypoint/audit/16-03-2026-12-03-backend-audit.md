## Assumptions used in this audit
- `main` is the review target and represents the deployable backend state.
- Straude is deployed on a multi-instance Next.js/Vercel-style runtime rather than a single long-lived process. This is an inference from the repo’s Vercel configuration and deployment docs.

## System understanding

Straude’s backend is a monolithic Next.js server app backed by Supabase. Public and session-authenticated HTTP entry points live under `apps/web/app/api/**`, with Supabase session clients used for user-scoped reads/writes and service clients used for elevated operations such as email lookups, admin analytics, signup-trigger backfills, referrals, and some notification/email flows.

Core data enters through two main write paths: CLI/web usage ingestion at `POST /api/usage/submit`, and social/content writes across posts, comments, follows, kudos, DMs, uploads, and prompts. Post visibility and most object-level access depend heavily on Supabase RLS and a few SECURITY DEFINER RPCs such as `get_feed`, `get_direct_message_threads`, and `lookup_user_id_by_email`. Async boundaries are mostly fire-and-forget side effects after the primary response: achievements, notifications, and email delivery. Storage is used for post images and DM attachments, with URLs later rendered directly by the web app.

Operationally, the backend has minimal built-in control points: no durable queue, no shared rate-limit store, and limited explicit logging around failures beyond console errors in a few paths. That keeps the stack simple, but it means trust-boundary mistakes around search, stored media URLs, or storage privacy have outsized impact.

## Findings

### BA-001: Public email search leaks private profiles
- Priority: P1
- Why it matters: the public search endpoint can return a private user record when the caller supplies an exact email address, which defeats the product’s documented “public profiles only” search boundary.
- Evidence:
  - `apps/web/app/api/search/route.ts:13-18` does not require authentication before serving search results.
  - `apps/web/app/api/search/route.ts:48-62` switches to the service client for `lookup_user_id_by_email` and then returns the matching `users` row without filtering `is_public`.
  - `apps/web/__tests__/api/search.test.ts:131-155` explicitly codifies that email lookup should return a private user (`is_public: false`).
  - `docs/API.md:75-85` documents `GET /api/search` as a public endpoint that “only returns public profiles.”
- Affected area: `GET /api/search`, `lookup_user_id_by_email`, user privacy/search discoverability
- Risk if shipped as-is: anyone who knows or guesses an email address can confirm that the person has a Straude account and retrieve their private-profile metadata surface, even though non-email search is documented and marketed as public-only.
- Recommended fix: remove the private-user return path from public search. Either require an authenticated, explicitly authorized “find by email” flow, or keep the RPC server-only but filter returned rows to `is_public = true` unless the caller is the same user or an admin-approved flow.
- Confidence: High

### BA-002: Stored media URLs can point at arbitrary HTTPS hosts while the app proxies any host
- Priority: P1
- Why it matters: post images and DM attachments can persist arbitrary external URLs, and `next/image` is configured to proxy any HTTPS hostname. Together this creates a broad server-side fetch surface and lets users inject externally hosted media into other users’ views.
- Evidence:
  - `apps/web/next.config.ts:5-7` allows `remotePatterns: [{ protocol: "https", hostname: "**" }]`.
  - `apps/web/app/api/posts/[id]/route.ts:106-114` accepts `images` as any array of URLs with no host/origin validation.
  - `apps/web/app/api/messages/route.ts:180-195` only validates the attachment object shape and count; it does not verify that `attachments[].url` belongs to Straude storage.
  - `apps/web/components/app/messages/MessagesInbox.tsx:544-552` renders image attachment URLs through `next/image`, and `apps/web/components/app/messages/MessagesInbox.tsx:564-570` exposes file attachment URLs directly to recipients.
  - `docs/reviews/review-20260221-233150-50c1c4.md:5-10` previously flagged the same wildcard image-host policy plus arbitrary stored post-image URLs as a high-severity security issue.
- Affected area: post editing, DM attachments, Next image optimization, recipient media rendering
- Risk if shipped as-is: attackers can store arbitrary external URLs that the app later fetches or renders on behalf of viewers. That expands SSRF/misconfiguration risk on the server side and enables recipient tracking or hostile content injection on the client side.
- Recommended fix: restrict stored media to first-party storage/CDN origins or stored object keys only, validate post/DM media URLs server-side before persisting, and tighten `remotePatterns` to known approved hosts.
- Confidence: High

### BA-003: DM attachment privacy is inconsistent and unsafe across environments
- Priority: P1
- Why it matters: the DM attachment storage model is contradictory across production/local setup, and the current production migration makes DM files publicly readable even though DMs are a private feature.
- Evidence:
  - `supabase/migrations/20260310000000_add_dm_attachments.sql:22-50` creates `dm-attachments` with `public = true` and a storage policy stating “Anyone can view dm attachments.”
  - `supabase/seed.sql:20-33` creates the same bucket with `public = false` during local resets.
  - `apps/web/app/api/upload/route.ts:175-184` always returns `getPublicUrl(...)`, assuming the attachment bucket is public.
  - `apps/web/components/app/messages/MessagesInbox.tsx:218-233` and `apps/web/components/app/messages/MessagesInbox.tsx:530-587` consume those returned URLs directly.
  - `docs/LOCAL_DEV.md:94-101` says `dm-attachments` is private “to match production expectations,” which is the opposite of the shipped migration.
- Affected area: DM attachments, Supabase storage policy, local/prod parity
- Risk if shipped as-is: in production, DM attachments are readable by anyone who obtains the URL; in local-reset environments, the same feature can silently break because the app still emits public URLs for a private bucket. The env mismatch also makes it easy to miss the privacy issue during local QA.
- Recommended fix: decide on one model and enforce it everywhere. For private DMs, make `dm-attachments` private in migrations and local seed, stop returning public URLs, and serve attachments via signed URLs or an authenticated proxy.
- Confidence: High

### BA-004: Documented rate limits are process-local only
- Priority: P2
- Why it matters: the app documents backend rate limits as a real control, but the implementation is just in-memory maps inside request handlers. That is not a dependable abuse control for a public production deployment.
- Evidence:
  - `apps/web/lib/rate-limit.ts:24-77` stores limiter state in process-local `Map`s.
  - `apps/web/app/api/auth/cli/init/route.ts:5-20` implements a separate process-local in-memory limiter for the unauthenticated CLI login-init route.
  - `docs/API.md:15-24` presents these limits as the backend’s rate-limiting model, including the public `POST /api/auth/cli/init` route.
  - `vercel.json:1-8` shows this app is configured for Vercel-managed runtime features; under that style of deployment, request handlers can run across multiple instances or cold starts. This is an inference from the repo configuration.
- Affected area: public CLI login init, uploads, usage submit, social writes
- Risk if shipped as-is: abusive clients can bypass the documented limits by spreading requests across instances or benefiting from instance churn. The most exposed path is unauthenticated CLI code creation, but the same weakness applies to social, upload, and usage endpoints.
- Recommended fix: move rate limiting to a shared backend primitive such as Redis/Upstash, Supabase-backed counters, or provider-native edge limits, and keep the in-memory helper only for tests/local fallback if needed.
- Confidence: Medium

### BA-005: Unsubscribe advertises one-click POST semantics but only implements GET and ignores update failures
- Priority: P3
- Why it matters: notification emails advertise one-click unsubscribe headers, but the backend endpoint only supports `GET` and always renders success even if the preference update fails.
- Evidence:
  - `apps/web/lib/email/send-comment-email.ts:64-68`, `apps/web/lib/email/send-direct-message-email.ts:48-52`, `apps/web/lib/email/send-empty-profile-email.ts:34-38`, `apps/web/lib/email/send-weekly-digest.ts:194-199`, and related senders emit `List-Unsubscribe-Post: List-Unsubscribe=One-Click`.
  - `apps/web/app/api/unsubscribe/route.ts:5-45` implements only `GET`; there is no `POST` handler for one-click unsubscribe requests.
  - `apps/web/app/api/unsubscribe/route.ts:17-26` does not inspect the update result before returning the success HTML page.
- Affected area: notification email unsubscribe flow
- Risk if shipped as-is: some email clients that honor one-click unsubscribe via `POST` will fail silently, and users can see a success confirmation even when the preference write did not persist.
- Recommended fix: add a `POST` handler for one-click unsubscribe, keep the current `GET` fallback, and check/log the Supabase update result before returning success.
- Confidence: Medium

## Release conditions / next actions

1. Fix the public exact-email search path so private profiles are not returned from `GET /api/search`.
2. Lock media persistence down to approved first-party storage and remove the wildcard image-host policy.
3. Choose and enforce a private DM attachment model consistently across migrations, local seed, upload responses, and client rendering.
4. If the current process-local rate limiting is kept for this release, accept that risk explicitly and document that the advertised limits are best-effort only; otherwise replace it with a shared limiter.
5. Repair the unsubscribe flow before the next email-focused release so advertised one-click unsubscribe semantics are truthful.
